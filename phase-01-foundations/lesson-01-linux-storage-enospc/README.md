## Objective

Diagnose a real `ENOSPC` failure by mapping an exact pathname to its backing filesystem and distinguishing data-capacity evidence from metadata-capacity evidence.

## Why this matters in production

Restarting a service or deleting files because one broad `df -h` result looked suspicious can destroy evidence, increase impact, or modify the wrong filesystem. The safe response is to verify the exact path, compare independent allocation resources, and predict how each observation changes the hypothesis ranking.

## Small mental model

A pathname is resolved component by component. Any directory in that path can be a mount point backed by a different filesystem. Creating a file requires the filesystem backing the exact target path to allocate every required resource; free data capacity on a broader path does not prove that every required allocation can succeed.

## Lab architecture and constraints

The setup creates one named disposable container with:

- no network or published ports;
- a read-only root filesystem;
- UID/GID `65534:65534` as the image, container, and interactive-shell default;
- all Linux capabilities dropped;
- `no-new-privileges` enabled;
- 0.5 CPU, 128 MiB memory, and 64 PIDs maximum;
- bounded in-memory filesystems totaling at most 17 MiB and owned by the unprivileged runtime user;
- no host bind mounts, Docker socket mount, credentials, or cloud access.

The failure is generated by the Linux kernel inside the disposable lab. It does not fill the Windows or WSL host disk.

- Cloud cost: zero.

- Host mounts and real secrets: prohibited.
